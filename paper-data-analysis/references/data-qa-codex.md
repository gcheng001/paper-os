# 路径 B：Codex 脏数据 6 步核查

> 本文件固化 Codex 脏数据 6 步核查流程 + 三条安全红线 + 技术保障。
> 方法论来源：Get 笔记《数据处理，用 Codex 按你的标准检查、修复》（快刀青衣课程）

---

## 核心逻辑

**不搞"AI 自动批量清洗"炫技，而是把人的业务经验转化为 AI 可执行的标准规则。**

| 理念 | 含义 |
|------|------|
| **标准由人定** | 业务口径、合格标准、例外场景由人定义 |
| **执行交给 AI** | AI 按标准批量扫描、识别问题、生成修复版 |
| **最终决策权在人** | 修复方案须经用户确认，AI 不擅自决策 |

**为什么不用"AI 自动清洗"**：脏数据的"脏"往往是业务语义层面的（如某个字段该填什么、某个空值该不该补），AI 无法凭数据本身判断，必须由人提供业务口径。

---

## 前置安全边界（3 条红线）

> 违反任意一条即**硬失败**，立即终止并报告用户。详见 [`../SKILL.md`](../SKILL.md)「三条安全红线」。

### 红线 1：只读优先

**第一阶段仅扫描盘点，禁止直接修改源文件。**

- 第一轮 Codex 调用**只读** CSV，输出问题清单
- 即使发现明显错误（如负数的年龄），也不在扫描阶段修复
- 修复动作必须在用户确认业务口径后（第 5 步）进行

**为什么**：没有业务口径确认前的"修复"可能改错（如负数年龄可能是数据录入错误，也可能是"未填写"的占位符）。

### 红线 2：仅操作测试副本

**所有数据操作必须在 `materials/_data_workspace/` 测试副本内进行，源文件路径只读。**

- Codex 工作目录强制设为 `materials/_data_workspace/`
- 源 CSV 复制进副本（`cp 源文件 _data_workspace/`）
- Codex 仅对副本操作，源文件路径通过权限控制只读

### 红线 3：禁止覆盖源文件

**质检清单/问题清单/修复版全部另存为新文件，修复版强制加 `_修复版` 后缀。**

- 质检清单 → `脏数据核查清单.md`（新建）
- 问题清单 → `问题清单.md`（新建）
- 修复版 → `原文件名_修复版.csv`（新建，**不改源**）

## 工作目录隔离（技术保障）

```
<论文项目>/
├── materials/
│   ├── 原始数据.csv              ← 源文件（只读，mtime 自检）
│   └── _data_workspace/          ← Codex 工作目录（强制）
│       ├── 原始数据_副本.csv      ← 副本（Codex 可读写）
│       └── ...（其他中间文件）
└── outputs/
    └── S-DATA-数据分析/
        ├── 脏数据核查清单.md
        ├── 问题清单.md
        ├── 原始数据_修复版.csv
        └── 检查日志.md
```

**Codex 启动指令**：工作目录参数强制设为 `materials/_data_workspace/`。

## mtime 自检（强制步骤）

**目的**：用文件系统的 mtime（修改时间）作为源文件未被改写的硬证据。

### 执行前

```bash
ls -la <源文件路径>
```

记录源文件 mtime（如 `Jun 27 09:30`）。

### 执行后

```bash
ls -la <源文件路径>
```

对比 mtime：

| 情况 | 处置 |
|------|------|
| mtime **未变化** | ✅ 通过，写入产物 frontmatter「源文件 mtime 未变」 |
| mtime **变化** | ❌ **硬失败**：源文件被改写，立即中止，报告用户，回退 |

## 六步实操流程

### 准备阶段：划定项目范围

**目标**：明确本次核查的边界，避免 Codex 越界操作。

**用户输入**：
- 待核查的数据文件夹路径（源文件所在）
- 项目背景（这是什么数据，业务语义）

**操作**：
1. 创建 `materials/_data_workspace/` 工作目录
2. 源 CSV 复制进副本：`cp <源文件> materials/_data_workspace/<源文件名>_副本.csv`
3. 执行前 `ls -la <源文件>` 记录 mtime
4. 通知用户：源文件已锁定只读，所有操作在副本进行

---

### 第一步：只读扫描识别

**目标**：仅扫描不修复不生成新文件，列出所有疑似问题。

**Codex 提示词要点**：

```
你是数据质量审查专家。请对 [_data_workspace/副本] 执行只读扫描。

严格要求：
1. 仅扫描，禁止修复任何数据
2. 禁止生成新文件
3. 禁止修改原文件

扫描维度：
- 字段命名不一致（同一含义不同命名，如"性别" vs "sex" vs "Gender"）
- 单位不统一（如身高既有 cm 又有 m）
- 数据类型异常（数字字段含文本）
- 缺失值分布（哪些字段缺失最多）
- 疑似异常值（超出合理范围）
- 重复记录
- 编码不一致（如性别 1/0 vs 男/女）

输出格式：每个问题标注
- 文件名
- 字段/位置
- 原文示例（3 条以内）
- 问题描述

扫描完成后，列出 5-8 个最影响后续校验的核心口径问题，向我提问确认。
不要自行假设业务口径。
```

**产出**：扫描报告（对话形式，暂不落盘）。

---

### 第二步：人机交互确认业务口径

**目标**：基于扫描结果，向用户提 5 个最影响校验的核心口径问题。

**提问原则**：
- 每问附**原文位置**（用户能直接定位）
- 每问给出 2-3 个候选答案供用户选择（降低用户认知负荷）
- 不问"开放式"问题，问"选择题"

**典型口径问题**：

| 问题类型 | 示例 |
|---------|------|
| 字段命名统一 | "性别字段存在'男/女'、'M/F'、'1/0'三种编码，应统一为哪种？" |
| 指标范围 | "年龄字段出现负数（-1, -99），是表示'未填写'还是录入错误？" |
| 缺失值规则 | "成绩为空的学生，应标记为'缺考'还是按 0 分处理？" |
| 单位统一 | "身高字段有 cm 也有 m，统一为哪个单位？" |
| 重复记录 | "同一学号出现多条记录，是保留最新一条还是合并？" |

**产出**：用户确认的 5 条业务口径（对话形式）。

---

### 第三步：生成可复用质检清单

**目标**：把用户确认的口径整理为可复用的《自检清单》，落盘到 `outputs/S-DATA-数据分析/脏数据核查清单.md`。

**清单格式**：

```markdown
# 脏数据核查清单

## 检查项 1：[检查内容，如"性别字段编码统一性"]

- **合格标准**：性别字段必须为"男"或"女"
- **严重等级**：必须通过
- **允许例外场景**：无

## 检查项 2：[检查内容]

- **合格标准**：[具体标准]
- **严重等级**：建议通过
- **允许例外场景**：[如有的话]
```

**严重等级枚举**：

| 等级 | 含义 | 处置 |
|------|------|------|
| **必须通过** | 不通过则数据不可用 | 必须修复 |
| **建议通过** | 影响分析质量但不致命 | 建议修复 |
| **可选** | 锦上添花 | 视时间而定 |

---

### 第四步：输出全量问题清单

**目标**：按质检清单逐项校验副本数据，生成《问题清单.md》。

**Codex 提示词要点**：

```
基于已确认的《脏数据核查清单》，对 [_data_workspace/副本] 执行全量校验。

要求：
1. 逐项对照清单的每个检查项
2. 严格只读，禁止修改数据，禁止生成修复版
3. 输出《问题清单.md》，每个问题包含：
   - 严重程度（必须通过/建议通过/可选）
   - 文件名
   - 字段/位置（行号）
   - 原文（实际值）
   - 问题描述
   - 建议修复（具体动作）
   - 是否需人工确认（是/否）
```

**问题清单格式**：

```markdown
## 问题 #1 [必须通过]
- **文件**：原始数据_副本.csv
- **位置**：第 23 行，"性别"字段
- **原文**：M
- **问题描述**：性别编码非"男/女"，违反检查项 1
- **建议修复**：M → 男
- **需人工确认**：否
```

**产出**：`outputs/S-DATA-数据分析/问题清单.md`

---

### 第五步：生成候选修复版

**目标**：严格按用户确认后的问题清单，生成 `_修复版` 文件。

**前置门禁**：用户须明示"可以修复了"或对问题清单逐项确认。

**Codex 提示词要点**：

```
基于已确认的《问题清单.md》，生成修复版数据。

严格要求：
1. 输出文件名：原始数据_修复版.csv（强制 _修复版 后缀）
2. 保留原文件的公式、格式、列结构
3. 仅按问题清单的建议修复执行，不自行扩展
4. 对"需人工确认"的问题，跳过不修复，在日志中标注
5. 修复完成后，写《检查日志.md》，记录：
   - 每个问题的处置（已修复/跳过/人工确认中）
   - 修复前后对照（前 10 条示例）
   - 未修复原因说明
6. 输出原文件与修复版的字段对照清单（确认结构一致）
```

**产出**：
- `outputs/S-DATA-数据分析/原始数据_修复版.csv`
- `outputs/S-DATA-数据分析/检查日志.md`

**执行后 mtime 自检**：
```bash
ls -la <源文件路径>
```
对比执行前 mtime，确认源文件未被改写。

---

### 第六步：沉淀项目专属 Skill（可选进阶）

**目标**：将本次核查流程提炼为项目内可复用技能。

**触发条件**：
- 同一类数据需定期核查（如每月问卷数据）
- 核查流程已跑通 2 次以上
- 用户明示"沉淀为技能"

**操作**：提示用户可将流程提炼为项目内 Skill（如 `.claude/skills/<项目名>-data-qa/SKILL.md`），复用本项目的质检清单与修复规则。

> 此步为可选进阶，不强制执行。

## 核心价值总结

| 价值 | 说明 |
|------|------|
| **标准由人定** | 业务口径不靠 AI 猜，靠人确认，避免"自动清洗"改错 |
| **执行交给 AI** | 人定标准后，AI 批量扫描识别问题，效率远超人工 |
| **安全可控** | 三条红线 + 工作目录隔离 + mtime 自检，源文件零风险 |
| **可复用** | 质检清单沉淀为项目资产，同类数据可重复使用 |
| **可追溯** | 问题清单 + 检查日志，每个修复动作有据可查 |
