## 三、核心原则速查

> 完整内容见 `soul.md`

| 原则 | 一句话 | 违反后果 |
|------|--------|---------|
| 人主导AI辅助 | 选题/观点/结论必须出自用户 | 学术不端 |
| 先研究后写作 | 没有认知差不动笔 | 论文无价值 |
| 当场解决问题 | 不带病进下一步 | 质量失控 |
| 幽灵文献零容忍 | 所有引用必须过S6核查 | 撤稿风险 |
| 输出去AI味 | S3/S4强制de-ai-polish | 被识别为AI生成 |
| 确认门禁 | AI不得自改status | 流程失控 |
| LOG.md同步更新 | 每步完成必须追加 | 工作无记录 |

## 五、断点续办检查清单

进入已有项目时，按以下顺序读取：

1. **PAPER.md** → 题目、研究问题、当前阶段
2. **LOG.md** → 工作日志
3. **state.json** → 权威断点状态
4. **outputs/** → 各步骤实际产物

**输出断点摘要**：
```
📌 论文：<题目>
✅ 已完成：S1-S3
⏳ 当前步骤：S4（讨论与结论）
📋 待确认：无
➡️ 下一步：继续S4 或 跳到S5自检
```

**红线**：不得重做已确认步骤。

## 六、de-ai-polish 集成规范

### 触发时机

| 步骤 | 强制/建议 | 说明 |
|------|----------|------|
| S1 选题报告 | 建议 | 选题报告非正式论文 |
| S3 章节初稿 | **强制** | 每节起草后立即调用 |
| S4 讨论结论 | **强制** | 输出前必须调用 |
| S5-S8 | 不强制 | 检查/核查类步骤 |

### Fallback 方案

如 `de-ai-polish` 不可用，手动执行以下检查：

1. ❌ 删除"随着XX的发展..."式宏观开头
2. ❌ 删除空洞升华结尾
3. ❌ 减少被动句使用
4. ❌ 删除信息密度极低的"句套子"
5. ✅ 检验方法：这句话能否删除而不影响意思表达？能删→删

## 九、常见问题FAQ

### Q1：能否跳步？

**可以**，论文场景允许带警告跳步。例如拿着已写完的论文直接进S5/S6。

**唯一例外**：S3起草动作要求S1选题已确认，或用户明示豁免。

跳步时输出：`⚠️ 前置步骤S<X>未完成，已跳过。后续如需回溯请随时告知。`

### Q2：子Skill被单独调用怎么办？

按"子Skill独立触发协议"执行：
1. 注入原则：读取 `soul.md`
2. 定位项目：有项目→检查前置产物；无项目→独立模式
3. 执行：按子Skill流程执行

### Q3：state.json 的 status 枚举？

- `pending` — 未开始
- `in_progress` — 进行中
- `pending_review` — 待用户确认（硬门禁）
- `confirmed` — 用户已确认（硬门禁翻转）
- `reported` — 已汇报（软确认步骤）

### Q4：产物文件的 frontmatter 规范？

每个产物头部带 YAML frontmatter：
```yaml
---
step_id: S<N>
status: pending_review | reported
key_conclusions:
  - 结论1（≤3条）
depends_on:
  - S<前置步骤>
---
```

下游步骤读取上游产物的 frontmatter 获取关键结论，读正文获取详细内容。
